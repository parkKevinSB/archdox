## User Direction And Strong Preferences

The user is actively shaping the product through review. Respect these
preferences:

- Do not hide old model incompatibility with compatibility fallback unless the
  user explicitly asks. Pre-production old reports/test reports may break.
- Do not preserve old report payloads at the cost of the new canonical model.
- Do not add broad abstractions just to make architecture look clean.
- Flower worker lanes should be execution-character based, not one worker per
  feature/harness.
- Flower steps must not block worker ticks with `Thread.sleep`, synchronous
  child-flow waits, or provider calls. Submit work and observe durable state,
  future completion, events, or `stepNo`.
- AI must be source-bounded and controlled. It should propose review findings,
  wording suggestions, or action candidates, not silently mutate law/source
  data or execute controlled actions.
- User-facing UI must use business language, not developer language such as
  `source-backed`, `dry-run`, provider code, or internal enum names.
- Mobile UI matters. Keep work screens compact and task-focused.
- If deploying, actually verify public URLs and container health.

## Agent Launcher And Downloads State

The latest phase focused on making local Agent installation less developer-like.

Implemented:

- `archdox-agent-launcher` now supports `--force-install`.
- Runtime environment passed from launcher to Agent now includes
  `CLOUD_API_BASE_URL`, so started runtimes connect to the configured Cloud API.
- Windows auto-start helpers exist under
  `archdox-agent-launcher/src/dist/windows/`.
- Top-level double-click helpers exist under `archdox-agent-launcher/src/dist/`:
  - `Install ArchDox Agent.bat`
  - `Remove ArchDox Agent.bat`
  - `README-WINDOWS.txt`
- `archdox-agent-launcher/build.gradle.kts` now creates a small Java runtime
  image with `jlink` and packages it into `jre/` inside the launcher zip.
- The release zip is now self-contained for normal Windows users. They should
  not need to install Java.
- The installer is not yet a signed EXE/MSI. It is a self-contained zip with a
  double-click `.bat` installer.
- `public-site/downloads/index.html` explains the new flow:
  1. download zip
  2. extract
  3. double-click `Install ArchDox Agent.bat`
  4. Windows logon auto-starts the Agent
  5. remove with `Remove ArchDox Agent.bat`

Important implementation detail:

- The `.bat` wrapper sets `JAVA_HOME` to the bundled `jre`.
- The scheduled-task runner also detects bundled `jre` and prepends it to
  `PATH`.
- The scheduled task is a current-user logon task named `ArchDox Agent`.
- It runs launcher `supervise` mode so the Agent restarts if the runtime process
  disappears or health is not confirmed.

Likely next step:

- Replace zip-plus-bat with a proper installer:
  - EXE/MSI installer wizard
  - Start Menu entry
  - uninstall registration
  - optional tray/settings UI
  - code signing
  - better pairing/setup UX

Do not jump straight into complex installer tech unless the user asks. The
current package is intentionally the first usable installer-like step.

## Construction Supervision Report And Checklist State

Large work before this handoff changed daily supervision reports toward a
structured checklist model.

Current intended model:

- `DAILY_LOG.dailyItems.groups[]` is canonical.
- A daily log group may be trade-based or phase-based.
- Trade-based hierarchy:
  - trade group
  - trade
  - sub-trade (`NONE` when absent)
  - work category / task group
  - process
  - inspection item
  - checklist rows
- Phase-based hierarchy:
  - phase/checklist kind
  - work category
  - process
  - review item
  - checklist rows if present
- Each row has structured result:
  - default/no value means not inspected / not selected
  - `COMPLIANT`
  - `NON_COMPLIANT`
  - row-level criteria/reference text
  - row-level action text
  - row-level photos
- The generated daily-log prose must be derived from structured rows unless
  user-facing "document wording" override exists.
- Do not reintroduce old `supervisionContent` as canonical input fallback.

Recent user intent:

- Inspection item itself can be a checklist row too, but generated prose should
  avoid duplicate repetition.
- If the inspection item row has a checked result, show it as the parent line.
- Child row lines should be indented/listed below.
- If the parent row is not selected, do not render duplicate parent text.

Potential follow-up:

- Continue verifying official reference PDF and JSON-derived catalog.
- Improve checklist docx outputs and daily-log preview/document parity.
- Keep UI labels Korean and avoid showing internal codes to normal users.
- Keep mobile report editor compact.

## Legal Review And AI Harness State

Current legal review direction:

- Legal corpus sync uses DB-backed law/reference material.
- Domain bindings provide stronger legal anchors than search candidates.
- Candidate-only references must not produce confident PASS.
- Legal review should remain source-backed and cautious.
- AI should not invent law names, article numbers, dates, or source versions.
- For daily-log review, AI may propose improved wording or missing supervision
  record language, but it should not claim real-world evidence exists beyond
  what user entered.

Recent quality direction:

- The user wants legal review to propose practical supervision-log wording, such
  as "KS 등 자재성능 관련 서류를 확인하였으며..." where the official checklist
  requires that the supervisor record such confirmation.
- The AI should not refuse to suggest such wording merely because it cannot
  personally verify the actual attached certificate. The qualified user is
  responsible for actual evidence.
- Avoid ending wording with awkward phrases like "...기록합니다." when normal
  report prose should say "...확인하고 첨부하였습니다."

Important boundary:

- Source-backed legal review is not final legal advice.
- Legal review findings should be structured and visible, not hidden in vague
  summaries.
- Automatic correction should only apply to fields with concrete replacement
  text, not generic instructions.

## Flower Worker And Async Rule

The user repeatedly corrected the design around Flower worker usage.

Current rule:

- Worker lanes should be few and based on execution character.
- Do not create a worker per harness/feature.
- Do not block Flower tick threads for slow I/O or child-flow completion.
- Use submit-and-observe patterns.
- Dedicated bounded executors are acceptable for external blocking work, but
  the Flower worker lane must not be held while waiting.

Known current lanes include:

- `document-io`
- `document-review`
- `report-preflight-review`
- `ai-harness`
- `monitoring`
- `platform-ops`
- `archdox-worker`
- `legal-sync`

Do not add a new lane casually.
