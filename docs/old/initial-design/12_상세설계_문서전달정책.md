# ArchDox 상세 설계 — 문서 전달 정책

## 1. 왜 별도 정책이 필요한가

원본 요구는 단순 다운로드가 아니라 **"사용자는 웹에서 진행 상태와 히스토리를 보고, 실제 문서는 이메일/사무소 내부 전달/출력으로 받는다"**에 가깝다.

따라서 ArchDox의 기본 원칙은 다음이다.

- 일반 사용자 화면에는 **문서 상태, 생성 이력, 전달 상태**만 노출한다.
- 실제 DOCX/PDF 파일은 기본적으로 **사무소 NAS 또는 개인 플랜 S3**에만 보관한다.
- 사용자가 파일을 직접 여는 기능은 `document_artifacts`를 그대로 노출하지 않고, **명시적 전달 요청(delivery request)** 으로만 처리한다.
- 모든 전달은 audit log와 만료 정책을 가진다.

## 2. 전달 채널

| 채널 | 대상 | 파일 위치 | 기본 정책 |
|---|---|---|---|
| `LOCAL_INBOX` | 사무소 플랜 기본 | 사무소 NAS | 완료 상태만 Cloud에 보고. 사무소 내부 폴더/관리 UI에서 수령 |
| `EMAIL_LINK` | 개인/사무소 선택 | 개인: S3, 사무소: 임시 암호화 패키지 또는 ArchDox Agent relay | 수신자 확인 + 만료 링크. 기본 만료 24h |
| `LOCAL_PRINT` | 사무소 플랜 | ArchDox Agent → 프린터 | 출력 로그만 Cloud에 보고 |
| `ADMIN_EXPORT` | Owner/Admin | S3 temp 또는 ArchDox Agent export | 해지/감사/백업용 zip. 별도 권한 필요 |
| `NO_DELIVERY` | 생성만 수행 | 저장소만 | 사용자는 `GENERATED` 상태만 확인 |

MVP 기본값:
- 사무소 플랜: `LOCAL_INBOX`
- 개인 플랜: `EMAIL_LINK` 또는 `NO_DELIVERY`
- 일반 사용자 웹에서 임의의 `GET /document-artifacts/{id}/download` 같은 API는 만들지 않는다.

## 3. 사무소 플랜 전달 흐름

```text
사용자 제출
  → Cloud: document_jobs 생성
  → ArchDox Agent: DOCX/PDF 생성, NAS 저장
  → ArchDox Agent: artifact logical ref + sha256 + bytes 보고
  → Cloud: GENERATED 상태 표시
  → 전달 요청이 있으면 document_delivery_requests 생성
```

### 3.1 `LOCAL_INBOX`

- ArchDox Agent가 `data/offices/{officeCode}/reports/{reportId}/output/`에 파일 저장
- Cloud에는 logical ref, 파일명, 해시, 크기만 저장
- 사무소 관리자/담당자는 Local Mgmt UI 또는 NAS 폴더에서 수령
- 사용자 앱에는 "문서 생성 완료 / 사무소 내부 보관함에 저장됨"만 표시

### 3.2 `EMAIL_LINK`

사무소 플랜에서 이메일 전달이 필요할 때는 두 모드를 둔다.

| 모드 | 설명 | 적합한 경우 |
|---|---|---|
| `LOCAL_SMTP` | ArchDox Agent가 사무소 SMTP 또는 로컬 메일 계정으로 직접 발송 | 파일이 Cloud를 거치면 안 되는 사무소 |
| `TEMP_ENCRYPTED_LINK` | ArchDox Agent가 PDF를 암호화 zip 또는 단일 PDF로 임시 S3에 업로드, Cloud가 메일 발송 | 외부 수신자에게 편하게 전달해야 하는 경우 |

`TEMP_ENCRYPTED_LINK`는 명시적 사용자/관리자 동의가 있을 때만 사용한다. 업로드 객체는 24시간 후 lifecycle로 삭제하고, Cloud DB에는 temp object key와 만료시각만 남긴다.

## 4. 개인 플랜 전달 흐름

개인 플랜은 ArchDox Agent가 없으므로 Cloud generator가 생성한 DOCX/PDF를 S3에 저장한다.

- 기본 저장소: S3 private bucket
- 다운로드/이메일 링크: CloudFront signed URL 또는 S3 presigned URL
- 링크 만료: 기본 24h, 재발급 가능
- 무료 플랜은 보존 기간/월간 생성 횟수를 제한한다.

## 5. 데이터 모델

```sql
document_delivery_requests (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  report_id BIGINT NOT NULL FK,
  job_id BIGINT FK,
  artifact_id BIGINT FK,
  channel TEXT NOT NULL,              -- LOCAL_INBOX | EMAIL_LINK | LOCAL_PRINT | ADMIN_EXPORT
  mode TEXT,                          -- LOCAL_SMTP | TEMP_ENCRYPTED_LINK 등
  status TEXT NOT NULL,               -- REQUESTED | PROCESSING | SENT | PRINTED | FAILED | EXPIRED | CANCELLED
  requested_by BIGINT FK users(id),
  recipient_email TEXT,
  recipient_name TEXT,
  delivery_ref TEXT,                  -- temp object key, printer job id, local outbox id 등
  expires_at TIMESTAMPTZ,
  sent_at TIMESTAMPTZ,
  error_code TEXT,
  error_message TEXT,
  created_at TIMESTAMPTZ NOT NULL,
  INDEX (office_id, report_id),
  INDEX (office_id, status, created_at DESC)
)
```

`document_artifacts`는 산출물의 정본 인덱스이고, `document_delivery_requests`는 산출물을 누구에게 어떤 방식으로 전달했는지의 이력이다.

## 6. API

```http
POST /api/v1/inspection-reports/{reportId}/deliver
GET  /api/v1/inspection-reports/{reportId}/deliveries
GET  /api/v1/document-deliveries/{deliveryId}
POST /api/v1/document-deliveries/{deliveryId}/cancel
```

요청 예:

```json
{
  "artifactType": "PDF",
  "channel": "EMAIL_LINK",
  "recipientEmail": "client@example.com",
  "recipientName": "홍길동",
  "message": "감리일지 전달드립니다."
}
```

사무소 플랜에서 ArchDox Agent 처리가 필요한 경우 Cloud는 `archdox_agent_commands`에 `DELIVER_DOCUMENT` 명령을 넣고, Agent가 완료/실패를 보고한다.

## 8. 보안 원칙

- 문서 파일 직접 다운로드 API는 기본 제공하지 않는다.
- 링크 기반 전달은 항상 만료시각, 1회성 토큰, audit log를 가진다.
- 외부 이메일 전달은 수신자, 요청자, artifact hash, IP, user-agent를 기록한다.
- 사무소 플랜에서 Cloud 임시 업로드를 쓰는 경우 파일은 암호화하고 lifecycle 삭제를 강제한다.
- 전달 실패와 재발송은 같은 delivery request를 갱신하지 않고 새 이력으로 남긴다.

## 9. 한 줄 요약

**`document_artifacts`는 파일 정본의 인덱스, `document_delivery_requests`는 파일을 밖으로 내보낸 행위의 감사 가능한 이력이다.** 사용자는 기본적으로 상태만 보고, 실제 문서 전달은 명시적 채널과 권한 정책을 거쳐 수행한다.
