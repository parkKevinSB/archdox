# ArchDox 상세 설계 — 문서 생성 파이프라인 & AI 추상화 계층

## 1. 사용자가 던진 요건

- 문서 자동화 생성 방식은 **아직 안 정함**. ChatGPT / 다른 AI / API 호출 등 무엇이든 붙일 수 있어야 함
- 문서 생성하려면 **문장도 만들어야** 하고, 문서 포맷(Word 등)도 있어야 함
- 문서 종류가 **많을 수 있고** 다 커버해야 함
- AI를 **붙일 수 있는 계층 구조**가 있어야 함 (어떤 AI를 쓸지는 미정)
- AI 안 쓰면 **"날것 그대로" 저장**도 지원해야 함

이 요건들은 **확장 가능한 plugin 구조**와 **field-level 옵션화**로 풀어야 한다.

## 2. 문서 생성의 4단계 (불변 파이프라인)

```
[Input]                  [Compose]                [Bind]                 [Render]
사용자 입력 +              필드 단위로                Template Engine        DOCX → PDF
사진/체크리스트   ─→      "이 필드는 어떻게  ─→     (docx4j) +      ─→    (LibreOffice
JSON payload              만들 것인가" 결정          field map              headless)
                          (Raw | Template | AI)
```

이 4단계는 **모든 문서 종류에 공통**이다. 다른 건 (a) input schema, (b) compose 정책, (c) DOCX 템플릿뿐.

### 2.1 Compose 단계가 핵심

"필드를 어떻게 채울지"를 결정하는 단계. 필드마다 다음 중 하나로 정해진다:

| Mode | 설명 | 예시 |
|---|---|---|
| `RAW` | 사용자가 입력한 그대로 | 날짜, 참석자 명단, 측정값 |
| `TEMPLATE` | 정형 문장 템플릿에 값 끼움 | "오늘 날씨는 ${weather}, 공정률은 ${progress}%입니다." |
| `AI_DRAFT` | AI가 초안 작성, 사용자 확정 | 작업사항 요약, 특이사항 정리 |
| `AI_REWRITE` | 사용자 메모를 AI가 정제 | "콘크리트 균열 발견 → '북측 벽체 콘크리트 표면에 폭 0.2mm 미세 균열이 관찰됨'" |
| `AI_SUMMARY` | 여러 입력을 합쳐 요약 | 체크리스트 50개 NG 항목 → 종합 의견 |
| `COMPUTED` | 다른 필드들에서 계산 | NG 개수 자동 합계 |

**중요**: AI 사용 여부는 **필드 단위로 결정**한다. 한 문서에서 일부 필드만 AI, 나머지는 RAW가 정상 케이스.

## 3. AI 추상화 계층

### 3.1 계층 구조

```
┌──────────────────────────────────────────────┐
│  Document Composer (도메인)                  │
│  필드별 Mode를 보고 적절한 Generator 호출    │
└──────────────────────────────────────────────┘
                  │
                  ▼ TextGenerator interface
┌──────────────────────────────────────────────┐
│  Generator Plug-ins                          │
│  ┌──────────┬──────────┬──────────┬───────┐  │
│  │ NoOp     │ Template │ OpenAI   │ ...   │  │
│  │ (raw)    │ (변수치환) │ (GPT-4)  │       │  │
│  ├──────────┼──────────┼──────────┼───────┤  │
│  │ Anthropic│ Gemini   │ Ollama   │ Bedrock│  │
│  │ (Claude) │          │ (local)  │ (AWS) │  │
│  └──────────┴──────────┴──────────┴───────┘  │
└──────────────────────────────────────────────┘
                  │
                  ▼
┌──────────────────────────────────────────────┐
│  Provider SPI                                │
│  - 인증/키 관리                              │
│  - rate limit / retry                        │
│  - cost metering                             │
│  - prompt cache                              │
└──────────────────────────────────────────────┘
```

### 3.2 핵심 인터페이스 (Java)

```java
package com.archdox.compose.ai;

public interface TextGenerator {
    GeneratorId id();           // openai-gpt-4o, anthropic-claude-4-7, noop, template-handlebars
    Capability capability();    // RAW | TEMPLATE | AI

    GenerationResult generate(GenerationRequest req);
}

public record GenerationRequest(
    String fieldKey,            // 어느 필드를 채우는가 (관측용)
    Mode mode,                  // RAW | TEMPLATE | AI_DRAFT | AI_REWRITE | AI_SUMMARY
    String promptTemplate,      // AI 모드에서 사용. ${field} placeholder 지원
    Map<String, Object> input,  // 사용자 입력 + 다른 필드 값
    List<String> stylingHints,  // "공식 문서체", "건축 용어 보존" 등
    int maxTokens,
    String tenantId,            // 비용 정산용 office_id
    String reportId             // trace
) {}

public record GenerationResult(
    String text,
    GeneratorId generator,
    int promptTokens,
    int completionTokens,
    long latencyMs,
    boolean confident,          // AI가 자체 판단 OK인지
    List<String> warnings       // "맥락 부족" "한자 변환 실패" 등
) {}

public interface GeneratorRegistry {
    TextGenerator resolve(GeneratorId id);
    List<GeneratorId> availableFor(String officeId);
}
```

### 3.3 등록 메커니즘

- Spring `@Component`로 구현체 등록 → `GeneratorRegistry`가 모음
- 새 AI 추가 = 새 `@Component` 클래스 하나. **도메인 코드 변경 없음**.
- application.yml로 활성화/비활성화:

```yaml
archdox:
  ai:
    enabled-generators:
      - noop
      - template-handlebars
      - openai-gpt-4o
      - anthropic-claude-sonnet-4-6
    default-for-mode:
      RAW: noop
      TEMPLATE: template-handlebars
      AI_DRAFT: anthropic-claude-sonnet-4-6     # 기본
      AI_REWRITE: anthropic-claude-haiku-4-5    # 가벼운 모델
      AI_SUMMARY: openai-gpt-4o
```

### 3.4 기본 제공 Generator (MVP 포함)

| ID | 역할 | 비용 | 비고 |
|---|---|---|---|
| `noop` | 입력값 그대로 반환 (escape만) | 0 | 항상 동작. AI 미사용 케이스의 디폴트. |
| `template-handlebars` | Handlebars로 변수 치환 (`{{weather}}`) | 0 | 정형 문장에 사용. AI 없이 자연스러운 문장 가능. |
| `template-jinja2` | 더 복잡한 분기/반복이 필요할 때 (선택) | 0 | Jinja-like 문법 |
| `compute-expression` | 산술/날짜 계산 (`sum(checklistNg)`) | 0 | Spring Expression Language 또는 자체 mini-DSL |

→ MVP는 위 4개만 있어도 **AI 없이 모든 문서 생성 가능**하다. AI는 옵션.

### 3.5 AI Generator 구현 (도입 시)

- `OpenAiTextGenerator` — official Java SDK 또는 직접 REST
- `AnthropicTextGenerator` — `anthropic-java`
- `OllamaTextGenerator` — 사무소 망 내 로컬 모델 (보안 민감 사무소용)
- `BedrockTextGenerator` — AWS Bedrock (한국 region 추가 시 검토)
- `GeminiTextGenerator` — Google Vertex AI

각 구현은 다음을 모두 처리:
- API key 주입 (per-office or platform)
- Rate limit (token bucket)
- Retry with backoff (`5xx`, `429`)
- Streaming은 옵션 (문서 생성은 일괄 처리라 sync로 충분)
- Prompt cache (같은 prompt 재사용 시 — Anthropic은 native 지원)

## 4. AI Key 관리 — BYOK (Bring Your Own Key) vs Platform-Managed

| 모델 | 누가 결제 | 키 저장 위치 | 적합한 케이스 |
|---|---|---|---|
| **Platform-Managed** | ArchDox가 결제 | Cloud Secrets Manager | 개인/소형 사무소, 사용량 한도 강제 |
| **BYOK** | 사무소 자체 결제 | 사무소가 입력, KMS 암호화 후 `office_ai_credentials` 테이블 | 대형 사무소, 자체 계정, 데이터 정책 |
| **Local LLM (Ollama)** | 사무소 자체 인프라 | ArchDox Agent 설정 | 보안 민감 사무소 (사진/도면이 외부 API에 노출 X) |

```sql
office_ai_credentials (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  generator_id TEXT NOT NULL,          -- openai-gpt-4o, anthropic-claude-..., ollama
  credential_kind TEXT NOT NULL,       -- API_KEY | OAUTH | NONE
  encrypted_secret BYTEA,              -- KMS-encrypted
  endpoint_url TEXT,                   -- self-hosted/Ollama용
  enabled BOOLEAN,
  monthly_cost_limit_usd NUMERIC,
  created_at TIMESTAMPTZ,
  UNIQUE(office_id, generator_id)
)

ai_call_logs (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  report_id BIGINT FK,
  field_key TEXT,
  generator_id TEXT NOT NULL,
  prompt_tokens INT,
  completion_tokens INT,
  cost_micro_usd BIGINT,              -- 0.000001 USD 단위
  latency_ms INT,
  cached BOOLEAN,
  status TEXT,                        -- OK | RATE_LIMITED | TIMEOUT | ERROR
  error_message TEXT,
  created_at TIMESTAMPTZ NOT NULL,
  INDEX (office_id, created_at DESC)
)
```

매 호출은 `ai_call_logs`에 기록 → 사무소별 월간 비용 산정 + admin 대시보드.

## 5. 보안/프라이버시 정책

- **AI에 보낼 입력에서 PII 자동 redact** 옵션 (사람 이름, 전화번호 등). 사무소 정책으로 켜고 끔.
- **사진/도면은 AI에 안 보냄이 기본**. 보내야 한다면 (Vision API) **별도 토글** + 사무소 명시 동의.
- **AI 응답은 항상 사용자가 확정** 후 저장. AI가 만든 문장을 자동 채택 X. (`AI_DRAFT`는 초안일 뿐)
- AI 사용/미사용은 **field-level 옵션**. 사무소가 "이 필드는 절대 AI 안 씀" 잠금 가능.

## 6. 문서 종류 카탈로그 (확장 가능)

문서 종류는 **DB row + DOCX 템플릿 파일 + JSON schema/compose/prompt 정책** 조합이다. 코드 변경 없이 추가 가능하다.

### 6.1 도메인 모델

```sql
document_types (
  id BIGSERIAL PK,
  type_code TEXT NOT NULL,             -- DAILY_SUPERVISION, PERIODIC_SAFETY,
                                       --  STRUCTURAL_DIAGNOSIS, COMPLETION_REPORT,
                                       --  CUSTOM_*
  display_name TEXT NOT NULL,
  category TEXT,                       -- 감리 | 점검 | 진단 | 준공 | 기타
  description TEXT,
  is_system BOOLEAN,                   -- true=기본 제공, false=사무소 커스텀
  office_id BIGINT,                    -- NULL이면 시스템 공통
  created_at TIMESTAMPTZ,
  UNIQUE(office_id, type_code)
)
```

`templates` 테이블의 정본 정의는 `02_상세설계_도메인및데이터.md` §3.4에 둔다. 핵심 컬럼은 `document_type_id`, `schema_json`, `compose_policy_json`, `ai_prompts_json`, `edit_status`이며, `09` 문서는 이 컬럼을 사용하는 compose 규칙만 설명한다.

### 6.2 시스템 기본 문서 종류 (예상)

- 감리일지 (일일/주간)
- 정기 안전점검 보고서
- 비정기 점검 보고서
- 구조 안전 진단 보고서
- 준공 검사 보고서
- 시정 지시서
- 회의록
- 사진대지

각 종류는 **버전 관리**. v1 → v2 마이그레이션 시 기존 리포트는 v1로 계속 생성. (`templates.version`)

## 7. Template Schema DSL

JSON Schema 기반 + ArchDox 확장 키워드.

```json
{
  "$schema": "https://archdox.app/schemas/template-v1.json",
  "templateCode": "DAILY_SUPERVISION_V1",
  "displayName": "감리일지 (일일)",
  "documentTypeCode": "DAILY_SUPERVISION",
  "fields": [
    {
      "key": "reportDate",
      "label": "감리일자",
      "type": "date",
      "required": true,
      "compose": {"mode": "RAW"},
      "binding": {"placeholder": "${reportDate}", "format": "yyyy-MM-dd"}
    },
    {
      "key": "weather",
      "label": "날씨",
      "type": "enum",
      "options": ["맑음", "흐림", "비", "눈", "안개"],
      "compose": {"mode": "RAW"}
    },
    {
      "key": "workSummary",
      "label": "작업 사항",
      "type": "text",
      "compose": {
        "mode": "AI_REWRITE",
        "fallback": "RAW",
        "promptRef": "rewrite-construction-log",
        "userConfirmationRequired": true
      },
      "binding": {"placeholder": "${workSummary}"}
    },
    {
      "key": "checklist",
      "label": "체크리스트",
      "type": "array",
      "itemSchema": {
        "category": {"type": "enum", "options": ["구조", "마감", "안전", "전기"]},
        "itemKey": {"type": "string"},
        "result": {"type": "enum", "options": ["OK", "NG", "NA"]},
        "remarks": {"type": "text"},
        "photos": {"type": "photoRef[]", "max": 4}
      },
      "binding": {"placeholder": "{{#each checklist}}...{{/each}}"}
    },
    {
      "key": "ngSummary",
      "label": "NG 항목 요약",
      "type": "text",
      "compose": {
        "mode": "AI_SUMMARY",
        "promptRef": "summarize-ng-items",
        "inputs": ["checklist"]
      }
    },
    {
      "key": "ngCount",
      "label": "NG 개수",
      "type": "integer",
      "compose": {
        "mode": "COMPUTED",
        "expression": "checklist[result=='NG'].length"
      }
    }
  ],
  "aiPrompts": {
    "rewrite-construction-log": {
      "system": "당신은 한국 건축 감리 문서를 작성하는 전문가입니다. 다음 메모를 공식 감리일지 문체로 정제하세요. 사실관계를 변형하지 말고, 용어는 한국 건축법령 표준을 따르세요.",

    "summarize-ng-items": {
      "system": "감리원이 발견한 NG 항목들을 종합 의견 1~3문장으로 정리하세요. 우선순위가 높은 항목을 앞에 두세요.",
```

이 schema 한 파일이 **(a) 입력 UI 자동 생성**, **(b) compose 정책**, **(c) DOCX 바인딩**, **(d) AI prompt**를 모두 담는다.

## 8. 클라이언트 UI 자동 생성

`05` 문서의 wizard는 **하드코딩된 화면이 아니라 schema-driven**이다.

```typescript
// client/web에서
const template = await api.templates.get(templateId);
return <WizardRenderer schema={template.schema} initialData={draft} />;
```

`WizardRenderer`는 schema의 `fields[]`를 순회하며:
- `type: date` → DatePicker
- `type: enum` → Select
- `type: text` + `compose.mode: AI_REWRITE` → Textarea + "AI로 정제" 버튼
- `type: array` → 반복 입력 카드
- `type: photoRef[]` → 사진 첨부 슬롯

→ 새 문서 종류 추가 시 **클라이언트 코드 0줄 변경**. schema 등록만으로 신규 wizard 자동 생성.

## 9. Compose 실행 흐름 (server-side)

`DocumentRenderFlow` (`03` §8.2) 안의 새 step `compose-fields`:

```java
@Component
public class ComposeFieldsStep extends Step {

    private final GeneratorRegistry registry;
    private final ComposePolicyResolver policyResolver;

    @Override
    protected StepResult onTick(StepContext ctx) {
        var report = ctx.payload(ReportPayload.class);
        var template = ctx.payload(TemplateSpec.class);
        var composed = new LinkedHashMap<String, Object>();

        for (var field : template.fields()) {
            var policy = policyResolver.resolve(field, ctx.officeId());
            var generator = registry.resolve(policy.generatorId());

            var req = new GenerationRequest(
                field.key(),
                policy.mode(),
                template.aiPrompts().get(policy.promptRef()),
                buildInput(field, report, composed),
                policy.stylingHints(),
                policy.maxTokens(),
                ctx.officeId(),
                ctx.flowId().flowKey()
            );

            try {
                var result = generator.generate(req);
                composed.put(field.key(), result.text());
                ctx.eventBus().publish(new FieldComposedEvent(
                    ctx.flowId(), field.key(), result));
            } catch (GeneratorException e) {
                if (policy.fallback() != null) {
                    var fallback = registry.resolve(policy.fallback());
                    composed.put(field.key(),
                        fallback.generate(req.withMode(Mode.RAW)).text());
                } else {
                    return StepResult.fail(e);
                }
            }
        }

        ctx.putState("composed", composed);
        return StepResult.done();
    }
}
```

`FieldComposedEvent`는 Bloom 이벤트로 발행되어 `ai_call_logs` 기록 + 진행률 SSE push에 사용.

## 10. AI 호출 위치 — Cloud vs ArchDox Agent

| 시나리오 | AI 호출 위치 | 근거 |
|---|---|---|
| 개인 플랜 + Platform-Managed AI | **Cloud API** | 입력이 Cloud에 있음, 단순 |
| 사무소 플랜 + Platform-Managed AI | **Cloud API** | 〃 |
| 사무소 플랜 + BYOK | **Cloud API** | 키만 사무소 것, 호출은 Cloud (감사/비용 통제) |
| 사무소 플랜 + Local LLM (Ollama) | **ArchDox Agent** | 데이터를 사무소 밖으로 안 내보냄 |
| 사무소 플랜 + "AI 사용 안 함" | NoOp/Template로 처리 | 둘 다 안 호출 |

ArchDox Agent에서 호출하는 경우 `compose-fields` step이 Local에서 돌도록 flow를 분기. (cloud-doc-generation orchestrator가 target 결정 시 `aiLocation`도 고려)

## 11. 실패 & Fallback 정책

| 케이스 | 정책 |
|---|---|
| AI provider 5xx / timeout | retry 3회 (exponential backoff) → 실패 시 `fallback` mode (보통 RAW) |
| 월 비용 한도 초과 | 즉시 fallback. 사용자에게 "이번 달 AI 한도 초과, 원본 그대로 사용됨" 안내 |
| API key 누락/만료 | template의 `fallback` 적용. admin 알림. |
| AI 응답이 PII 검출 룰에 걸림 | redact 후 재시도, 그래도 걸리면 fallback |
| 응답이 비어있음/너무 짧음 | 사용자에게 "AI가 충분한 응답을 못 했습니다" 표시 + 원본 노출 |

→ **항상 RAW로 떨어질 수 있어야 한다**. AI는 augmentation이지 dependency가 아니다.

## 12. 관측

- Bloom 이벤트: `FieldComposedEvent`, `AiCallFailedEvent`, `AiCostThresholdExceededEvent`
- Micrometer 지표: `archdox.ai.call.duration{generator,office,status}`, `archdox.ai.cost{generator,office}`
- Admin 대시보드: 사무소별 월 비용, generator별 성공률, 평균 latency
- 사용자 화면: 리포트 생성 후 "AI가 N개 필드 보조, 비용 ~원" 표시 (BYOK일 때만)

## 13. MVP 범위 / 후속

**MVP 포함:**
- 4단계 파이프라인 + `compose-fields` step
- `noop`, `template-handlebars`, `compute-expression` 3개 generator
- field-level mode 결정 (schema)
- AI는 1개 provider만 (Anthropic Claude 권장 — 한국어 품질 + prompt cache)
- BYOK 미포함 (Platform-Managed only)

**Phase 8 이후:**
- BYOK + KMS
- 추가 provider (OpenAI, Gemini, Ollama)
- Vision/사진 분석 통합
- 사용량 기반 과금
- PII redact

## 14. 한 줄 요약

**문서 생성은 "필드별 Mode 결정 → Generator 호출 → DOCX 바인딩"의 plugin 구조.** AI는 `TextGenerator` interface 뒤로 추상화되어, 어떤 AI든 `@Component` 하나로 붙는다. AI 없이도 `noop`/`template`만으로 모든 문서가 생성된다. 새 문서 종류는 schema + DOCX 등록만으로 추가, 클라이언트 코드 변경 0.
