# ArchDox 상세 설계 — 템플릿 편집기 (Format Builder)

## 1. 사용자가 던진 요건

- 나중에는 **문서 포맷을 쉽게 만들 수 있는 로컬 프로그램** 또는
- **웹뷰 상에서 원하는 문서 포맷을 수정**할 수 있는 도구가 필요
- 포맷에서 필요한 **데이터/필드와 연결**해서 UI에서 체크/입력 가능하게

이건 사실 두 가지 요건이다:
1. **DOCX 같은 시각적 포맷을 만드는 도구** (디자인 영역)
2. **그 포맷의 필드와 입력 schema를 매핑하는 도구** (데이터 바인딩)

ArchDox에서는 둘을 분리해서 다룬다.

## 2. 두 가지 편집기, 한 가지 정의

> 사용자가 만든 모든 템플릿은 결국 `templates` row 하나 = (DOCX 파일 + `schema_json` + `compose_policy_json` + `ai_prompts_json`) 묶음이다. 데이터 모델 정본은 `02_상세설계_도메인및데이터.md` §3.4에 둔다.

이 묶음을 만드는 **편집기는 2종**:

| 편집기 | 위치 | 대상 사용자 | 특징 |
|---|---|---|---|
| **Cloud Template Studio** | 웹 (admin 또는 사무소 owner 권한) | 본사 콘텐츠 팀, 사무소 IT 담당 | 시스템 기본 템플릿 + 사무소 커스텀 템플릿 모두 |
| **Local Template Editor** | ArchDox Agent 내장 페이지 | 사무소 내부, 보안 민감 | 사무소 NAS에만 보관하는 비공개 템플릿 |

두 편집기는 **동일한 schema 포맷**(`09` §7)을 produce한다. Local에서 만든 걸 나중에 Cloud로 export/import 가능.

## 3. 핵심 워크플로 — "Word에서 만들고, 웹에서 묶는다"

DOCX의 시각 디자인은 Word/한글이 압도적이다. 직접 만들 필요 없다. 대신:

```
[Step 1] Word에서 평소처럼 문서 디자인
         ↓ placeholder를 ${field}/{{field}} 형식으로 적기
[Step 2] DOCX 업로드
[Step 3] 시스템이 placeholder 자동 추출
[Step 4] 추출된 placeholder ↔ schema 필드 시각적 매핑
[Step 5] 필드별 compose Mode 지정 (RAW/TEMPLATE/AI_*)
[Step 6] (AI 사용 필드만) prompt 작성
[Step 7] 미리보기 (샘플 데이터로 즉시 렌더)
[Step 8] 저장 = 새 templates 버전
```

이 흐름이면 **사무소 직원도 1~2시간 교육으로 자체 템플릿 제작 가능**. 디자이너/개발자가 필요 없다.

### 4.2 기능 목록

| # | 기능 | 설명 |
|---|---|---|
| 4.1 | DOCX 업로드 | drag&drop, placeholder 자동 추출 (`${...}`, `{{...}}` 둘 다 인식) |

| 4.3 | 필드 추가/삭제/순서 | UI에서 schema 편집 |

| 4.5 | enum 옵션 편집 | 한 줄에 하나씩 / CSV import |
| 4.6 | 배열 필드 | 반복 placeholder `{{#each items}}...{{/each}}` 도우미 |
| 4.7 | compose Mode 설정 | 드롭다운 + Mode별 상세 패널 |
| 4.8 | AI prompt 편집기 | Markdown 에디터. `${input.field}` 자동완성. 토큰 카운터. |
| 4.9 | 샘플 데이터 입력 | JSON 또는 폼 입력 |
| 4.10 | 즉시 미리보기 | 샘플 데이터 + 현재 schema/policy로 임시 렌더 → DOCX/PDF 다운로드 |
| 4.11 | AI 비용 견적 | "이 템플릿을 1회 생성 시 평균 ~원" (provider 토큰 단가 기반) |
| 4.12 | 검증 (Lint) | 미매핑 placeholder, 미사용 필드, 순환 참조, AI prompt에 정의 안 된 변수 |
| 4.13 | 버전 관리 | save 시 `templates.version` 증가. 기존 진행 리포트는 옛 버전으로 계속 생성 |

| 4.16 | 권한 | 시스템 템플릿은 admin만, 사무소 템플릿은 office OWNER/ADMIN |
| 4.17 | 게시 | DRAFT → PUBLISHED 상태. PUBLISHED만 일반 사용자가 wizard로 사용 |

### 4.3 데이터 모델 기준

`templates`와 `template_revisions`의 컬럼 정의는 `02` 문서 §3.4를 정본으로 삼는다. Template Studio는 그 모델을 다음 규칙으로 사용한다.

- 저장 버튼은 항상 `template_revisions` row를 먼저 만들고, 성공하면 `templates` current row를 갱신한다.
- `edit_status=PUBLISHED` 이후 파괴적 변경은 같은 row를 수정하지 않고 새 `version`을 만든다.
- `sample_data_json`은 preview와 lint에만 사용하고 실제 리포트 payload와 섞지 않는다.
- `storage_kind=LOCAL` 템플릿은 Cloud에 DOCX 원본을 올리지 않고 metadata와 logical ref만 동기화한다.

## 5. Placeholder 추출 알고리즘

DOCX 업로드 시 docx4j로 본문/표/헤더/푸터를 walk:

```java
public Set<PlaceholderRef> extract(Path docxPath) {
    var doc = WordprocessingMLPackage.load(docxPath.toFile());
    var visitor = new PlaceholderVisitor();
    new TraversalUtil(doc.getMainDocumentPart().getContent(), visitor);
    return visitor.placeholders(); // {name, location: BODY|HEADER|FOOTER|TABLE_CELL,
                                   //  block: SIMPLE|EACH|IF, lineNumber}
}
```

지원 표기:
- `${fieldKey}` — 단순 치환
- `${fieldKey | format:yyyy-MM-dd}` — 포맷터
- `{{#each checklist}}...{{/each}}` — 반복 (Handlebars 호환)
- `{{#if showSection}}...{{/if}}` — 조건
- `${photo:photoId, size:medium}` — 사진 삽입 (size: thumb|medium|original)

추출된 placeholder는 schema field와 자동 매핑 시도 (동명). 매칭 안 되는 것만 사용자가 수동 매핑.

## 6. AI Prompt 편집기

별도 컴포넌트:
- `system` / `user` 분리 입력
- `${input.foo}` 자동완성 (현재 schema 기반)
- `| filter:result=NG`, `| toBulletList` 같은 helper pipe 문서화
- "테스트 실행" → 샘플 데이터로 generator 호출 → 결과 + 토큰/비용 표시
- 여러 prompt를 비교 (A/B) 후 채택

## 7. 미리보기 파이프라인

```
[샘플 데이터] + [현재 schema] + [현재 DOCX 업로드본]
       ↓
   compose-fields (AI 호출 가능, 비용 추적)
       ↓
   docx4j 바인딩
       ↓
   LibreOffice headless PDF
       ↓
   브라우저 새 탭에서 보기 + 다운로드
```

미리보기는 별도 short-lived job. 실제 `document_jobs`와 다른 테이블 또는 `job_type=PREVIEW`로 표시.

## 8. Local Template Editor

Cloud Studio와 동일 UI지만:
- ArchDox Agent의 `/admin/templates` 페이지 (`06` §4.3에 추가)
- 템플릿이 ArchDox Agent의 `data/offices/{code}/templates/`에만 저장됨
- Cloud DB의 `templates` row에는 metadata + `storage_kind=LOCAL`만 기록
- 미리보기는 ArchDox Agent에서 직접 렌더 (Cloud 왕복 없음)

→ 사무소 자체 표준 양식, 영업 비밀 양식, 관공서 비공개 양식 등 보관용.

## 9. Schema 마이그레이션

기존 리포트가 v1 schema로 작성 중인데 v2가 published되면?

- **진행 중인 draft**: v1 계속 사용 (안 깨짐)
- **신규 draft**: v2 사용
- **재생성(regenerate) 요청**: v1 vs v2 선택 모달
- **schema breaking change**: edit_status가 DRAFT일 때만 자유롭게 가능. PUBLISHED 이후 destructive change는 새 버전 강제.

Breaking change 자동 감지 (Lint):
- 필드 제거
- type 변경
- enum option 제거
- required 추가

## 10. 권한 매트릭스

| 액션 | Personal | Office MEMBER | Office ADMIN | Office OWNER | Platform Admin |
|---|---|---|---|---|---|
| 시스템 템플릿으로 리포트 작성 | ✓ | ✓ | ✓ | ✓ | ✓ |
| 사무소 커스텀 템플릿 보기 | - | ✓ | ✓ | ✓ | ✓ |
| 사무소 커스텀 템플릿 편집 | - | - | ✓ | ✓ | ✓ |
| 사무소 커스텀 템플릿 게시 | - | - | - | ✓ | ✓ |
| 시스템 템플릿 편집 | - | - | - | - | ✓ |

## 11. UI ↔ Field 연결 보장

`09` §8에서 wizard는 schema-driven이라고 했다. 이걸 깨지 않으려면 편집기에서도 **schema-first**:

- 사용자가 DOCX placeholder만 추가하고 schema에 등록 안 하면 Lint 에러
- 사용자가 schema field만 추가하고 DOCX에 안 쓰면 경고 (의도적인 경우 OK)
- compose Mode가 AI_*인데 prompt 미정의면 Lint 에러
- 필드 type 변경 시 wizard preview 자동 갱신

→ 편집기에서 "샘플 wizard 미리보기" 버튼이 있어서, 실제 사용자가 보는 입력 화면도 즉시 확인 가능.

## 12. MVP 범위

구현 순서는 `08_구현순서_및_기능목록.md` Phase 6에 포함한다.

**MVP 포함:**
- DOCX 업로드 + placeholder 자동 추출
- schema 트리 편집 (type 5종: date/text/number/enum/array)
- compose Mode 4종 (RAW / TEMPLATE / COMPUTED / AI_DRAFT)
- 단일 AI prompt 편집
- 샘플 데이터 미리보기 (DOCX/PDF 다운로드)
- DRAFT/PUBLISHED 상태
- 시스템 템플릿 admin 편집만

**Phase 8 이후:**
- 반복/조건 placeholder (Handlebars 전체)
- 사무소 커스텀 템플릿 편집 UI
- Local Template Editor
- A/B prompt 비교
- 사진 삽입 placeholder 시각화
- Schema diff & 마이그레이션 도구

## 13. 한 줄 요약

**Word로 디자인 → 업로드 → placeholder를 schema 필드와 시각적 매핑 → compose Mode와 AI prompt 지정 → 미리보기 → 게시.** 새 문서 종류 추가에 개발자 손이 안 닿게 만드는 게 목표. Cloud 웹 편집기와 ArchDox Agent 내장 편집기 두 종, 동일 schema 포맷.
