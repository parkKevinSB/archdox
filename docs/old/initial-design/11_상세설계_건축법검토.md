# ArchDox 상세 설계 — 건축법 검토 & 법령 연결 (Regulation Engine)

## 1. 사용자가 던진 요건

- 작성 중 UI에서 **자동으로 건축법 관련 검토**
- 체크박스 체크 여부, 특이사항 입력 내용 검사
- 저장/제출 시점에 **오늘 감리 내용 검토** → 법 위반/누락 자동 감지
- **건축법 항목을 별도로 보유**해서 UI의 박스/체크박스에 항목별 매핑
- 화면 표시 옵션: 하단 노출 / 아이콘 옆 작은 (i) 클릭 → 팝업 / 또는 더 나은 방식
- **이 기능은 필수**

→ "법령은 별도 코퍼스로 관리하고, UI 필드/체크리스트 항목에 코드로 매핑하고, 작성/저장/제출 3단계에서 룰 + AI로 검토한다"가 답.

## 2. 4축 설계

```
[1] Regulation Catalog          [2] Field ↔ Regulation Mapping
    건축법 코퍼스                    템플릿 필드/체크 항목과 법령 연결
         │                              │
         └──────────────┬───────────────┘
                        ▼
         [3] Review Engine
         Rule + AI 검토 (작성/저장/제출 시점)
                        │
                        ▼
         [4] UX Surface
         인라인 (i) 아이콘, 사이드 패널, 검토 보고서
```

각 축을 독립 모듈로 둔다. 법령 데이터가 바뀌어도 UI/검증 엔진은 안 흔들리고, AI provider가 바뀌어도 매핑은 안 흔들린다.

## 3. Regulation Catalog (법령 코퍼스)

### 3.1 데이터 모델

```sql
regulation_sources (
  id BIGSERIAL PK,
  source_code TEXT NOT NULL UNIQUE,    -- BUILDING_ACT, BUILDING_ACT_ENFORCEMENT,
                                       --  STRUCTURAL_STANDARDS, FIRE_SAFETY_STANDARDS,
                                       --  ENERGY_STANDARDS, KCS, KDS, MOLIT_NOTICE_*,
                                       --  LOCAL_ORDINANCE_*
  display_name TEXT NOT NULL,
  authority TEXT,                      -- 국토교통부, 행정안전부, 지자체
  effective_from DATE,
  effective_to DATE,
  source_url TEXT,                     -- law.go.kr 등 원문 링크
  jurisdiction TEXT                    -- KR | KR-SEOUL | ... (지자체 조례)
)

regulation_articles (
  id BIGSERIAL PK,
  source_id BIGINT NOT NULL FK,
  article_code TEXT NOT NULL,          -- "건축법§49" / "건축법시행령§51-③" 등
  article_no TEXT,                     -- "49"
  paragraph TEXT,                      -- "③"
  title TEXT,
  body_text TEXT,                      -- 원문 전체
  body_html TEXT,                      -- 강조/링크 포함 표시용
  summary TEXT,                        -- 1~2문장 핵심 요약 (수동 또는 AI 작성)
  keywords TEXT[],                     -- 검색용
  applies_to TEXT[],                   -- ["RESIDENTIAL", "PUBLIC", ...]
  building_use_codes TEXT[],           -- 용도분류 코드
  related_codes TEXT[],                -- 다른 조문 ref (예: KDS, KCS)
  version INT NOT NULL,
  effective_from DATE,
  effective_to DATE,
  status TEXT NOT NULL,                -- ACTIVE | SUPERSEDED | REPEALED
  embedding VECTOR(1536),              -- pgvector, AI 검색용 (옵션)
  created_at TIMESTAMPTZ NOT NULL,
  UNIQUE(source_id, article_code, version),
  INDEX (article_code),
  INDEX (status)
)

regulation_clauses (
  -- 조문 안의 더 잘게 쪼갠 단위 (호/목)
  id BIGSERIAL PK,
  article_id BIGINT NOT NULL FK,
  clause_path TEXT NOT NULL,           -- "1.가.(2)"
  body_text TEXT NOT NULL,
  summary TEXT,
  UNIQUE(article_id, clause_path)
)

regulation_aliases (
  -- 약칭/별칭 (관습적 표현 → 정식 코드)
  alias TEXT PRIMARY KEY,
  article_id BIGINT NOT NULL FK
)
```

### 3.2 카탈로그 입수 방법

| 출처 | 방법 | 비고 |
|---|---|---|
| 국가법령정보센터 (law.go.kr) | OpenAPI (`/DRF/lawService.do`) 정기 fetch | 무료, 신청 필요 |
| 국토부 고시 | PDF/한글 다운로드 → OCR/HWP parsing | 자동화 어려움, 수동 + AI 보조 |
| KCS/KDS | 한국건설기술연구원 자료실 | PDF 위주, 수동 등록 |
| 지자체 조례 | 자치법규정보시스템 (elis.go.kr) | 지역별 |

→ **MVP는 핵심 법령(건축법, 건축법시행령, 건축물의 구조기준규칙, 건축물의 피난·방화구조 등의 기준에 관한 규칙) 정도만 시드**하고, 점진 확장.

### 3.3 갱신 정책

- 법령 개정은 잦다. `regulation_articles.version` + `effective_from/to`로 시점별 버전 관리.
- 사용자 리포트 작성 시 **그 리포트의 기준일(`reportDate`)에 유효한 버전**을 보여준다.
- 갱신은 admin 콘솔의 "법령 카탈로그 관리"에서 (`07` §6.2에 추가).
- 변경 알림: `RegulationUpdatedEvent` Bloom → 영향받는 활성 템플릿/사무소에 알림.

## 4. Field ↔ Regulation Mapping

### 4.1 매핑은 schema 안에 선언

`09` 문서 §7의 template schema에 `regulationRefs` 필드 추가:

```json
{
  "key": "fireSafetyCheck",
  "label": "방화구획 설치 확인",
  "type": "boolean",
  "compose": {"mode": "RAW"},
  "regulationRefs": [
    {"code": "건축법§49", "relevance": "PRIMARY", "summary": "건축물의 피난·방화구조 등"},
    {"code": "피난방화구조규칙§14", "relevance": "DETAIL"},
    {"code": "건축법시행령§46", "relevance": "DETAIL"}
  ],
  "validations": [
    {
      "kind": "REQUIRED_WHEN",
      "when": "buildingUseCode in ['BUSINESS','ASSEMBLY','MULTI_FAMILY']",
      "message": "해당 용도 건축물은 방화구획 확인이 필수입니다 (건축법§49)"
    }
  ]
}
```

### 4.2 매핑 종류

| 매핑 | 의미 | 효과 |
|---|---|---|
| 필드 → 단일 조문 | "이 체크박스는 §49와 직결" | (i) 아이콘 / 호버 / 팝업으로 조문 표시 |
| 필드 → 다중 조문 | "이 항목은 본법 + 시행령 + 규칙" | 사이드 패널에 묶음 표시 |
| 체크리스트 카테고리 → 조문 그룹 | "구조 카테고리 항목들 = KDS 14xx" | 카테고리 헤더에 (i) 표시 |
| 자유 텍스트 필드 → 검색 인덱스 | "특이사항 내용을 법령 코퍼스에 검색해 관련 조문 자동 제안" | AI/embedding 기반 추천 |
| 조건부 매핑 | "건물 용도가 X일 때만 §Y 적용" | validations 조건절 평가 |

### 4.3 매핑 관리

- 매핑은 **schema의 일부**이므로 템플릿 편집기(`10` 문서)에서 편집한다.
- 편집기에 "법령 추가" 버튼 → `regulation_articles` 검색 모달 → 코드 선택 → relevance 지정
- 사무소 커스텀 템플릿도 동일 매핑 사용 가능 (자체 조례 등은 사무소가 직접 추가)

## 5. Review Engine (검토 엔진)

### 5.1 검토 시점

| 시점 | 트리거 | 무엇을 검사 | 비용 |
|---|---|---|---|
| **편집 중 (live)** | 필드 blur, debounce 1s | 그 필드 한정 룰 (RULE_VALIDATION) | 0 (룰만) |
| **단계 저장** | step 저장 API | 해당 step의 필드들 + 누락 점검 (RULE) | 0 |
| **제출 (Submit)** | submit API | 전체 룰 + AI 검토 (REVIEW_FULL) | AI 호출 1~3회 |
| **수동 검토 요청** | 사용자가 "지금 검토" 버튼 | 현재까지 입력분 전체 + AI 검토 | AI 호출 |

→ 편집/저장은 **즉시 + 무료**, 제출/수동은 **AI 포함 + 비용**. 사용자가 비용을 인지하고 호출하는 구조.

### 5.2 룰 종류 (RuleEngine, AI 불필요)

```java
public sealed interface RegulationRule {
    RuleId id();
    String articleCode();
    RuleScope scope();      // FIELD | STEP | REPORT
    List<RuleViolation> evaluate(ReportSnapshot snapshot);
}

record RequiredWhenRule(...) implements RegulationRule {}
record MutuallyExclusiveRule(...) implements RegulationRule {}
record MinPhotoCountRule(...) implements RegulationRule {}
record DateOrderRule(...) implements RegulationRule {}
record NumericRangeRule(...) implements RegulationRule {}
record EnumChoiceRule(...) implements RegulationRule {}
record CrossFieldConsistencyRule(...) implements RegulationRule {}
```

룰은 **데이터 + 코드**. 단순한 룰은 schema의 `validations` JSON으로 선언적으로(SpEL), 복잡한 룰은 Java `@Component`로 등록.

### 5.3 AI 검토 단계 (REVIEW_FULL)

```
[1] 룰 검토 (즉시, 비용 0)
    └→ RuleViolation 목록
[2] 관련 법령 후보 추출 (embedding 또는 keyword)
    └→ 사용자 입력(특히 자유 텍스트)에서 키워드/임베딩 → top-K 법령 조문
[3] AI 검토 호출 (Anthropic Claude 권장 — 한국 법령 한국어 품질)
    ├ 입력: 리포트 전체 + 관련 법령 조문 본문 + 룰 결과
    └ 출력 JSON: [
        { severity: ERROR|WARN|INFO,
          articleCode: "건축법§49",
          fieldKey: "fireSafetyCheck",
          message: "방화구획 확인이 표시되지 않았으나 ...",
          suggestion: "...",
          confidence: 0.8 }
      ]
[4] 결과 통합 → ReviewReport
[5] 사용자에게 인라인 표시 + 제출 차단/경고
```

AI prompt 표준 템플릿(요약):
```text
[SYSTEM]
당신은 대한민국 건축법 검토 전문가입니다. 아래 감리/점검 보고서 초안과
관련 법령 조문을 검토하여, 누락·위반·확인 필요 사항을 JSON으로 보고하세요.
없는 사실을 추론하지 말고, 제공된 법령과 보고서 데이터만 근거로 하세요.

[USER]
## 보고서 데이터
${reportJson}

## 관련 법령 조문
${articlesJsonl}

## 룰 검사 결과
${ruleViolationsJson}

응답 형식: {"findings": [...]} (위 명세)
```

### 5.4 사후 검토 (선택)

`document_jobs`의 `GENERATED` 이후 별도 비동기 flow로 **품질 점수**(quality score) 산정 가능. 사무소 대시보드에 "이번 달 평균 완성도 92%" 같은 지표.

## 6. UX — 어떻게 보여줄까

사용자 질문: 인라인 / 팝업 / 사이드 패널 — 답은 **세 가지를 다 쓴다, 단 layer별로**.

### 6.1 Layer A: 필드 옆 (i) 아이콘 (Always-on, Static)

- 매핑된 법령이 있는 필드는 라벨 옆에 작은 (i) 아이콘 표시
- 호버 → 1줄 summary 툴팁
- 클릭 → 우측 사이드 패널에 조문 본문 + 출처 링크
- 모바일: 아이콘 tap → 바텀시트
- 키보드 접근성: 포커스 시 ARIA 라이브 영역

### 6.2 Layer B: 사이드 패널 / 바텀시트 (Reference Panel)

- 화면 우측 상시 토글 가능한 "관련 법령" 패널
- 현재 작성 중인 step의 모든 매핑 조문을 카드 리스트로
- 검색창 (법령 코퍼스 전체 검색)
- 카드 = 조문코드 + 1줄 요약 + 펼침 시 원문 + "원문 보기" 링크

### 6.3 Layer C: 인라인 인디케이터 (Live Validation)

- 룰 위반/경고는 필드 아래에 빨강(ERROR) / 노랑(WARN) / 파랑(INFO) 줄로 즉시 표시
- 메시지 끝에 "법령 §X 참조" 링크 → 사이드 패널에 해당 카드 highlight

### 6.4 Layer D: 제출 전 검토 보고서 (Review Report)

- "제출하기" 버튼 옆에 "지금 검토" 버튼
- 클릭 시 AI 검토 호출 (비용 안내 모달 — BYOK 케이스)
- 결과 모달:
  - ERROR N건 (제출 차단 옵션 — 사무소 정책에 따라)
  - WARN N건
  - INFO N건 (참고 사항)
  - 각 finding: 어느 필드/어느 조문/제안 문구
  - "이대로 제출" / "수정하기" / "AI 제안 적용"
- 제출 후에도 `report.review_reports`에 보관, 감사용

## 7. 데이터 모델 (검토 결과)

```sql
review_reports (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  report_id BIGINT NOT NULL FK,
  trigger TEXT NOT NULL,               -- AUTO_ON_SUBMIT | MANUAL | POST_GENERATION
  rule_engine_version TEXT,
  ai_generator_id TEXT,
  status TEXT NOT NULL,                -- RUNNING | DONE | FAILED
  total_findings INT,
  error_count INT,
  warn_count INT,

  id BIGSERIAL PK,
  review_report_id BIGINT NOT NULL FK,
  source TEXT NOT NULL,                -- RULE | AI
  severity TEXT NOT NULL,              -- ERROR | WARN | INFO
  field_key TEXT,
  article_code TEXT,
  message TEXT NOT NULL,
```

## 9. Flower Flow — ReviewFlow

```text
FLOW_TYPE = "report-review"
FLOW_KEY  = reportId + ":" + reviewReportId

steps:
  1. snapshot-report
  2. run-rule-engine
  3. (optional) retrieve-relevant-articles    # embedding/keyword 검색
  4. (optional) call-ai-reviewer
  5. merge-findings
  6. persist-review-report
  7. publish-review-completed                 # Bloom: ReviewCompletedEvent
```

`submit` 시점에 자동 실행 + 수동 트리거. `cloud-doc-generation` flow의 사전 step으로 endpoint policy에 따라 끼울 수도 있음 (예: "ERROR 있으면 생성 차단").

## 10. 검토 정책 (사무소 설정)

```yaml
office_review_policy:
  on_submit_run_review: true
  block_on_error: true                  # ERROR 있으면 제출 차단
  block_on_warn: false
  ai_review_enabled: true
  ai_review_max_monthly: 500
  ai_generator: anthropic-claude-sonnet-4-6
  retrieval_top_k: 8
  pii_redact_before_ai: true
```

사무소가 비용/정책에 맞게 조정. 개인 플랜은 simpler 기본값.

## 11. 보안 / 정확성

- **AI는 보조이지 판결자가 아니다.** 모든 finding에 "AI 의견 — 최종 확인은 책임 기술자가" 면책 문구.
- 법령 원문 표시는 **출처 + 시점 + 버전** 명시. 사용자가 어느 시점 법령을 기준으로 검토받았는지 추적 가능.
- AI에 보내는 데이터는 PII redact 가능 (`09` §5)
- 룰 엔진과 AI는 **공급망 검증된 법령 코퍼스만** 사용. 사무소가 임의 텍스트를 "법령"이라고 등록하지 못하게 시스템 catalog는 admin only.

## 12. 향후 확장

- 사무소별 **자체 체크리스트 표준** + 법령 매핑 (대형 사무소가 자사 quality 기준 추가)
- 지자체 **조례 자동 적용** (프로젝트 주소의 시군구로 자동 적용 룰셋 선택)
- **이미지 분석** 결합: 사진에서 균열 검출 → 자동 위반 후보 생성 (`08` 문서 Phase 8 AI와 연계)
- **법령 변경 영향 분석**: 새 개정안이 published되면 진행 중 프로젝트의 영향 조문 자동 알림
- **외부 시스템 연계** (세움터 등) — 보안 리스크 신중 검토

## 13. MVP 범위 (Phase 6과 함께)

## 14. 한 줄 요약

**법령은 별도 코퍼스(`regulation_articles`)로 관리하고, 템플릿 schema에서 필드/체크 항목과 `articleCode`로 매핑한다.** 룰 엔진은 즉시·무료로 검사하고, AI는 제출/수동 시점에 비용을 인지하고 호출한다. UI는 (i) 아이콘 + 사이드 패널 + 인라인 인디케이터 + 제출 전 검토 보고서 4-layer로 보여준다. 새 법령/룰/매핑 추가는 데이터/schema 변경만으로 가능, 도메인 코드 변경 없음.
