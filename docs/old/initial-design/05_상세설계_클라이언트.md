# ArchDox 상세 설계 — 클라이언트 (Web/iOS/Android)

## 1. 사용자 요청 정리

- 웹 UI 또는 어플리케이션
- iOS / Android 모두 동작
- 카메라로 촬영하면서 감리/안전점검 체크, 특이사항 입력, 임시 저장
- 임시 저장은 API 서버에서

## 2. 결정: PWA-first + Capacitor 래핑

사용자 앱의 시각/레이아웃 방향은
`docs/architecture/CLIENT_PRODUCT_UI_DIRECTION.md`를 따른다. 초기 리뷰 결과
기존 `admin` 콘솔의 UI가 더 적합하다는 피드백이 있었으므로, 실제 사용자용
`client/web`도 ArchDox admin과 같은 제품군으로 보이게 만든다. 즉, 다크
사이드바, 밝은 작업영역, 흰 패널, 컴팩트한 버튼/리스트/상태 배지를 공유한다.
다만 일반 사용자는 관리자 화면의 복잡한 운영 기능을 그대로 보지 않고,
프로젝트/리포트/사진/문서 생성 흐름을 더 단순한 작업 화면으로 사용한다.

### 2.1 비교

| 옵션 | 장점 | 단점 |
|---|---|---|
| **PWA-first (React)** + Capacitor 래핑 | 코드베이스 1개, 카메라/오프라인/푸시 PWA 가능, 즉시 배포 | iOS PWA의 일부 제약 (background sync 불완전), Capacitor 래핑 시 약간의 native 빌드 비용 |
| Flutter | 단일 코드, 성능 좋음 | 한국 폼/문서 입력 위젯 정제도 ↓, 웹 출력은 SEO/성능 보통, 학습 비용 |
| React Native | 풍부한 native 모듈 | iOS/Android 둘 다 native 빌드/사이닝 필요, 웹은 별도 |
| Native 2개 (Swift + Kotlin) | 최고 성능/UX | 코드 2배, 의미 없음 (입력+사진+조회 앱) |

**결정 근거**: 이 앱은 폼 입력 + 사진 + 조회 중심이고, native UX 요구가 크지 않다. PWA로 95%를 처리하고, App Store/Play 배포가 필요한 시점에만 Capacitor로 native shell을 입혀 카메라/푸시/딥링크 권한을 보강한다.

### 2.2 구성

```text
client/
  apps/
    web/              # Vite + React 18 + TS
                      # 빌드 산출물 = PWA (manifest + service worker)
    mobile/           # Capacitor 6 wrapper
                      # web 빌드를 동일하게 로드
                      # iOS/Android 프로젝트 (Xcode/Android Studio)
  packages/
    ui/               # shadcn/Tailwind 기반 컴포넌트
    api-client/       # OpenAPI 생성 client (Cloud API)
    offline-store/    # IndexedDB 기반 draft store
    media/            # 카메라/EXIF/resize 유틸 (workers)
```

## 3. 핵심 화면

용어 기준:
- `Project`: 최상위 프로젝트/계약/업무 묶음
- `Site`: 현장. 하나의 프로젝트 안에 여러 현장이 들어갈 수 있다.
- `Inspection Target`: 현장 아래의 건축물, 층, 구역, 설비, 부재, 자재,
  작업구역 등 실제 점검/촬영/문서화 대상

화면 우선순위 (개발 순):
1. 로그인 / 회원가입 (소셜 옵션은 후순위)
2. 사무소 선택 (멤버십 ≥ 2면 표시, 1개면 자동)
3. 프로젝트 목록 / 상세
4. 프로젝트 안의 현장(site) 목록 / 상세
5. **감리일지 작성 wizard** ⭐ (핵심)
   - Step 1: 헤더 (날짜, 날씨, 참석자, 공정률)
   - Step 2: 작업 사항 (자유 텍스트 + 정형 항목)
   - Step 3: **체크리스트 + 사진** (반복)
   - Step 4: 특이사항 / 지시사항
   - Step 5: 검토 후 제출
6. 사진 상세 (썸네일 그리드, 클릭 시 작업본 fetch)
7. 리포트 상태 / 진행률 / 알림
8. 정기 안전점검 wizard (감리일지와 별도 템플릿)
9. 설정 (알림, 캐시, 오프라인 모드)

## 4. 카메라 & 입력 UX

### 4.1 촬영 흐름
1. 체크리스트 항목 옆 "사진" 버튼 → `<input type="file" capture="environment" accept="image/*">` 사용
2. 촬영 즉시 클라이언트 worker에서:
   - EXIF 읽기 (날짜/GPS) 후 EXIF strip (개인정보 보호 정책에 따라)
   - canvas로 long-edge 1920px resize → JPEG 0.85 (T1)
   - long-edge 320px webp 0.7 (T2 썸네일)
3. `/photos/intent` 호출 → 응답대로 업로드
4. 업로드는 background queue. 사용자는 다음 사진/항목으로 즉시 이동 가능

사진 업로드 인증 경계:
- 개인 플랜과 사무소 기본값(`CLOUD_MEDIATED`)은 S3 presigned POST/PUT을 사용한다.
- 사무소 직접 업로드(`ARCHDOX_AGENT_DIRECT`)는 Cloud가 발급한 짧은 만료 signed upload token을 사용한다.
- PWA/브라우저에 사무소별 mTLS client certificate를 설치하지 않는다. mTLS는 ArchDox Agent와 Cloud API 사이의 service 인증에만 사용한다.

### 4.2 망 끊김 대응 (오프라인-first)
- IndexedDB에 `draft_steps`, `pending_photos`, `pending_intents` 큐
- 화면에는 "동기화 대기 N개" 인디케이터
- Service Worker `BackgroundSync` API + 폴링 fallback (iOS는 BG sync 미지원 → 앱 열릴 때 flush)
- 충돌 정책: 서버 `updated_at` vs 로컬 — last-write-wins (단, 다른 사용자 변경이 있으면 confirm 모달)
- 서버 draft 저장 모드는 office policy를 따른다. 기본값은 Cloud 암호화 저장(`CLOUD_ENCRYPTED`)이고, 보안 민감 사무소는 ArchDox Agent가 상세 payload 정본을 갖는 `LOCAL_ONLY` 모드다.
- 클라이언트는 두 모드를 구분해 UI를 바꾸지 않는다. 저장 API는 동일하고, 응답의 `syncState`만 `SAVED`, `QUEUED_LOCAL_ONLY`, `CONFLICT`처럼 표시한다.

### 4.3 모바일 특수 케이스
- iOS PWA: 카메라 권한, 파일 시스템 접근 OK. 푸시는 iOS 16.4+에서 PWA 푸시 지원 (manifest + APNs 등록). 안 되면 Capacitor로 래핑.
- Android: PWA 푸시 충분. installability prompt 적극 표시.
- 화면 회전/태블릿: tablet portrait 화면도 wizard 단계 유지 (양쪽 패널 X)

## 5. 인증 & 권한
- JWT access (15분) + refresh (14일, httpOnly cookie 또는 SecureStorage)
- Capacitor 환경에서는 `@capacitor/preferences` + `@capacitor-community/keychain` 사용
- 사무소 전환 → `X-Office-Id` 헤더 갱신 + 캐시 invalidate

## 6. 상태 관리
- 서버 상태: TanStack Query (캐시 + invalidation)
- 클라이언트 상태: Zustand (현재 office, wizard step)
- 폼: react-hook-form + zod schema
- 실시간: SSE 구독 (`/events/stream`) → query invalidation에 연결

## 7. 빌드 / 배포
- Web PWA: Vite build → CloudFront + S3
- iOS: Capacitor build → Xcode archive → App Store Connect
- Android: Capacitor build → AAB → Google Play
- 모두 동일 web build를 공유. native shell만 다름.
- 버전 정책: web은 hot ship, native shell은 web의 protocol-min-version 만 검사 (호환 끊기지 않게)

## 8. 디바이스 권한 & UX 규칙
- 카메라 권한은 "처음 사진 찍기" 누를 때 요청 (즉시 요청 X)
- GPS는 옵션. 기본 OFF. 켜기 토글은 설정에서.
- 알림 권한은 첫 리포트 제출 직후 자연스러운 시점에 요청
- 백그라운드 업로드 실패 시 알림 (Capacitor LocalNotifications)

## 9. 접근성 / i18n
- 한국어 기본. en 초벌 번역 옵션
- 폼 라벨, 버튼 명확. 60대 이상 사용자 고려 → 글자 크기/대비 옵션
- 다크 모드는 후순위
