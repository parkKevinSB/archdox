# ArchDox 상세 설계 — 도메인 모델 & 데이터

## 1. 도메인 경계 (Bounded Context)

abyss-runner의 패키지 분할 원칙과 동일하게, **기능 단위로 패키지를 자른다**. 운영 로직과 도메인 로직은 섞지 않는다.

```text
com.archdox
  account          # 계정, 인증, 약관, 탈퇴
  office           # 사무소(=테넌트), 멤버십, 플랜
  project          # 최상위 프로젝트/계약/업무 묶음
  site             # 현장(site), 프로젝트 안의 물리적 장소
  inspectionTarget # 건축물/층/구역/설비 등 점검 대상 트리
  inspection       # 감리일지, 정기점검 - draft, step, submit
  photo            # 사진 메타, 분류, 라벨 피드백
  document         # 문서 생성 job, artifact 인덱스
  template         # DOCX 템플릿 관리
  agent       # agent 등록, heartbeat, command pull
  license          # feature entitlement, usage counter
  notification     # email/push 발송
  audit            # 감사 로그
  operations       # admin 전용 use case (도메인 패키지 안 침범)
  global           # security, config, error, flower/bloom bootstrap
```

각 패키지 내부 layering:
```
{feature}
  controller      # HTTP 입출력, DTO 변환만
  application     # use case (서비스), Bloom listener
  domain          # 엔티티, value object, repository interface
  workflow        # Flower flow factory, step (장기 작업만)
  event           # Bloom 이벤트 클래스 (과거형)
  infra           # JPA 구현, 외부 시스템 어댑터
```

## 2. 멀티테넌시 모델

### 2.1 핵심 결정
- **단일 DB, 단일 스키마, `office_id` row-level 격리**.
- 대부분의 비즈니스 테이블은 `office_id NOT NULL`을 가진다. 예외는 명시적으로 둔다: `offices`, `users`, `office_memberships`, `feature_codes`, `plans` 같은 글로벌 정본과, 시스템 공통 템플릿/문서종류처럼 `office_id NULL`이 "플랫폼 기본값"을 뜻하는 테이블.
- 보안 한층 강화 시 PostgreSQL **RLS(Row Level Security)** 정책을 추가하고, JPA 세션 시작 시 `SET app.current_office_id = ?`를 박는다. (MVP는 application layer filter만)

### 2.2 사용자–사무소 N:M
사용자는 여러 사무소에 속할 수 있다. 개인 사용자도 자동 생성된 `personal-{userId}` 사무소에 속한다.

```sql
office_memberships (
  id,
  user_id,
  office_id,
  role,           -- OWNER | ADMIN | MEMBER | VIEWER
  status,         -- ACTIVE | INVITED | SUSPENDED | LEFT
  joined_at,
  UNIQUE(user_id, office_id)
)
```

JWT claim에는 `user_id`만 박고, `office_id`는 요청 헤더 `X-Office-Id` 또는 URL prefix (`/api/v1/offices/{officeId}/...`)에서 받아 매 요청마다 membership 검증한다. abyss-runner의 인증/권한 검사 패턴과 동일.

### 3.2 프로젝트, 현장(site), 점검 대상 & 리포트

큰 틀은 `project`다. `project`는 고객 계약, 감리/점검 업무 묶음,
반복 관리 패키지처럼 비즈니스 단위의 최상위 컨테이너다. 하나의
프로젝트 안에는 여러 `site`(현장)가 있을 수 있다.

`site`는 현장의 영어 표현이다. 건설 현장, 건축물 안전점검 현장,
시설/캠퍼스/지점처럼 실제 점검과 사진 촬영이 발생하는 물리적 장소를
뜻한다.

그 아래에는 `inspection_target` 트리가 붙는다. 예를 들면:

```text
project
  site: A 현장
    target: 101동 BUILDING
      target: 3층 FLOOR
        target: 전기실 ROOM
          target: 배전반 EQUIPMENT
```

MVP의 기존 `projects` 테이블은 일부 현장성 필드(`address`,
`building_type`)를 갖고 있지만, 장기 설계의 의미는
`project -> site -> inspection_target -> report/document`다. 다음 단계에서
여러 현장/여러 건축물/여러 설비를 다뤄야 하면 `sites`,
`inspection_targets`, `inspection_report_targets`를 별도 테이블로 추가한다.

```sql
projects (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  name TEXT NOT NULL,
  address TEXT,
  building_type TEXT,           -- 공동주택 / 단독 / 근생
  start_date DATE,
  end_date DATE,
  status TEXT NOT NULL,         -- ACTIVE | ARCHIVED
  created_by BIGINT FK users(id),
  created_at TIMESTAMPTZ NOT NULL,
  INDEX (office_id, status)
)

-- Future: 프로젝트 안의 물리적 현장
sites (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  project_id BIGINT NOT NULL FK,
  site_code TEXT,
  name TEXT NOT NULL,
  address TEXT,
  site_type TEXT,               -- CONSTRUCTION | BUILDING_SAFETY | FACILITY | OTHER
  start_date DATE,
  end_date DATE,
  metadata_json JSONB,
  status TEXT NOT NULL,         -- ACTIVE | ARCHIVED
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL,
  INDEX (office_id, project_id, status)
)

-- Future: 현장 아래의 건축물/층/구역/설비/자재/작업구역 등
inspection_targets (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  project_id BIGINT NOT NULL FK,
  site_id BIGINT NOT NULL FK,
  parent_target_id BIGINT FK,
  target_type TEXT NOT NULL,    -- BUILDING | FLOOR | ROOM | ZONE | EQUIPMENT | ...
  code TEXT,
  name TEXT NOT NULL,
  address TEXT,
  metadata_json JSONB,
  status TEXT NOT NULL,
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL,
  INDEX (office_id, site_id, target_type)
)

inspection_reports (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  project_id BIGINT NOT NULL FK,
  -- Future: primary_site_id BIGINT FK,
  report_no TEXT NOT NULL,
  report_type TEXT NOT NULL,    -- DAILY_SUPERVISION | PERIODIC_SAFETY | ...
  title TEXT,
  status TEXT NOT NULL,         -- DRAFT | STEP_SAVED | READY_TO_GENERATE
                                --  | GENERATION_REQUESTED | GENERATING
                                --  | GENERATED | DELIVERED | FAILED | CANCELLED
  current_step TEXT,
  template_id BIGINT FK templates(id),
  archdox_agent_id BIGINT FK,     -- 사무소 플랜만, 개인은 NULL
  requested_by BIGINT FK users(id),
  generated_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL,
  updated_at TIMESTAMPTZ NOT NULL,
  UNIQUE(office_id, project_id, report_no),
  INDEX (office_id, status, updated_at DESC)
)

-- Future: 리포트가 여러 점검 대상을 다룰 때 사용
inspection_report_targets (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  report_id BIGINT NOT NULL FK,
  target_id BIGINT NOT NULL FK,
  role TEXT NOT NULL,           -- PRIMARY | SECONDARY | REFERENCE
  snapshot_json JSONB NOT NULL, -- 생성 시점의 target 이름/주소/path 스냅샷
  UNIQUE(report_id, target_id)
)

inspection_report_steps (
  id BIGSERIAL PK,
  report_id BIGINT NOT NULL FK,
  step_code TEXT NOT NULL,      -- ex) HEADER, WEATHER, ATTENDEES, CHECKLIST, REMARKS
  payload_storage_mode TEXT NOT NULL DEFAULT 'CLOUD_ENCRYPTED',
                                -- CLOUD_ENCRYPTED | LOCAL_ONLY
  payload_json JSONB,           -- 검색/표시 가능한 비민감 요약 또는 개발용 fixture
  payload_ciphertext BYTEA,     -- Cloud 암호화 저장 모드의 full payload
  local_draft_ref TEXT,         -- LOCAL_ONLY일 때 ArchDox Agent draft logical key
  payload_hash BYTEA,           -- 정합성 검증용 sha256
  client_revision INT DEFAULT 0,
  saved_by BIGINT FK,
  saved_at TIMESTAMPTZ NOT NULL,
  UNIQUE(report_id, step_code)
)

inspection_checklist_items (
  -- 체크리스트 항목별 결과를 별도 테이블로
  id BIGSERIAL PK,
  report_id BIGINT NOT NULL FK,
  category TEXT,
  item_key TEXT,
  result TEXT,                  -- OK | NG | NA
  remarks TEXT,
  photo_ref_count INT DEFAULT 0,
  INDEX (report_id)
)
```

Draft payload 정본 정책:
- 기본값은 `CLOUD_ENCRYPTED`다. PWA 오프라인 동기화, 멀티기기 이어쓰기, 고객지원/복구를 위해 Cloud가 full payload를 암호화해 보관한다. DB에서 바로 읽히는 평문 JSON을 정본으로 두지 않고, application-level encryption 또는 KMS envelope encryption을 사용한다.
- 보안 민감 사무소는 `LOCAL_ONLY`를 선택할 수 있다. 이때 Cloud에는 `step_code`, `saved_at`, `payload_hash`, `local_draft_ref`만 남기고, 상세 payload는 ArchDox Agent의 draft store가 정본이다.
- 두 모드 모두 클라이언트 API 계약은 동일해야 한다. 서버가 어느 저장 모드를 쓰는지는 office policy로 결정하고, 클라이언트는 저장 성공/동기화 상태만 본다.

### 3.3 사진 & 미디어

```sql
photos (
  id BIGSERIAL PK,
  office_id BIGINT NOT NULL FK,
  report_id BIGINT FK,                -- 리포트와 미연결 사진 허용

  hash_sha256 BYTEA,                  -- 중복 업로드 합치기
  storage_kind TEXT NOT NULL,         -- AGENT_MANAGED | S3_IA | S3_GLACIER | DELETED
  storage_ref TEXT NOT NULL,          -- AGENT_MANAGED: logical key (path는 ArchDox Agent가 해석)
                                      -- S3_*: bucket/key
  thumbnail_s3_key TEXT,              -- 모든 케이스 공통, 작은 webp
  ai_top_categories JSONB,            -- [{label,confidence}, ...]

  created_at TIMESTAMPTZ NOT NULL,
  INDEX (office_id, report_id),
  INDEX (hash_sha256)
)

photo_label_feedback (
  -- AI 재학습용 (MVP는 수집만)
  id BIGSERIAL PK,
  photo_id BIGINT NOT NULL FK,
  predicted_top1 TEXT,
  confirmed_label TEXT,
  confirmed_by BIGINT FK users(id),
  confirmed_at TIMESTAMPTZ NOT NULL
)
```

**중요**: Cloud DB의 `photos.storage_ref`는 NAS의 실제 절대경로를 절대 저장하지 않는다. logical key (예: `report-12345/checklist-77/photo-1`)만 저장하고, ArchDox Agent가 자기 설정의 `nas_root + officeCode + key`로 실제 경로를 해석한다. (`11` 보안 원칙)

## 5. abyss-runner 대비 차이 — 의도적 결정

- abyss-runner는 단일 게임 인스턴스라 office 개념이 없다. ArchDox는 모든 비즈니스 테이블에 `office_id`를 강제한다.
- abyss-runner의 `operations` 패키지는 콘텐츠/모니터링 어드민. ArchDox는 `operations`가 **테넌트 관리** + `support` + `monitoring` + `contents`로 더 무겁다 → 별도 admin SPA로 분리 (`07` 문서).
- abyss-runner는 단일 JVM. ArchDox는 Cloud API와 ArchDox Agent **두 개 JVM이 동일 이벤트 모델(Bloom)을 다른 bus 인스턴스로 사용**한다. Cross-JVM은 REST/WebSocket으로 변환, 내부는 Bloom 그대로. (`03` 문서)
