# ArchDox 상세 설계 — Local Document Agent & 관리 UI

## 1. 사용자가 던진 두 질문

1. "로컬서버도 스프링부트로 짤까요" → **YES (이유 §2)**
2. "사내 로컬서버 관리는 C# WinForms? WebForms? 알아서 결정" → **ArchDox Agent 내장 React 페이지(localhost) (이유 §4)**

## 2. 왜 Spring Boot인가

| 기준 | Spring Boot | C# .NET (WinForms/WPF) | C# .NET 서비스 + WebForms |
|---|---|---|---|
| Cloud API와 도메인/이벤트 모델 공유 | ⭕ Bloom/Flower/엔티티/DTO 그대로 | ❌ 양쪽 코드 이중화 | ❌ |
| 멀티 OS (Windows 사무소 + Linux NAS) | ⭕ JVM 어디서나 | ❌ Windows only (WinForms) | ❌ |
| 패키징 / 서비스 등록 | jpackage/launch4j + Windows Service / systemd | 좋음 | 보통 |
| docx4j / LibreOffice headless | 풍부 | 별도 라이브러리 (Open XML SDK) | 별도 |
| 운영 인력 학습 | 같은 팀이 둘 다 안다면 동일 | C# 별도 인력 | C# 별도 |

ArchDox는 **Cloud API의 도메인 코드(엔티티/이벤트/Flower step)를 일부 공유**하므로 Spring Boot로 통일하는 게 가장 비용 효율적이다. abyss-runner 패턴(Bloom/Flower) 재사용도 결정 근거.

C# 옵션은 다음 경우 재검토:
- 사무소가 Windows 환경에 Native UI를 강하게 요구 (트레이 아이콘, 폴더 셸 연동 등)
- Hancom HWP 자동화처럼 Windows COM 의존 기능이 핵심이 될 때

→ 이 경우라도 **메인 Agent는 Spring Boot, 윈도우 specific helper만 C#으로 분리**하는 게 낫다.

## 3. ArchDox Agent 구성

### 3.1 패키지

```text
com.archdox.agent
  bootstrap          # Spring Boot main, pairing wizard, install token
  config             # 설정 (NAS path, printer, AI endpoint)
  pairing            # 초기 Cloud 연결 / mTLS 인증서 발급
  pull               # commands long-poll
  heartbeat          # 5s ~ 60s interval
  workflow           # Flower flow (DocumentRenderFlow 등)
  document           # docx4j 어댑터, PDF converter
  photo              # NAS 저장, 썸네일, hash
  nas                # SMB/CIFS/NFS/local mount adapter
  printer            # 프린터 큐
  internal           # /internal/v1/* (mgmt UI 호출)
  webui              # mgmt UI static + REST
  ai                 # Python AI client
  store              # SQLite local index (JOOQ or JPA)
  global             # Bloom/Flower 부트스트랩 (Cloud와 공유 모듈 사용)
```

### 3.2 설치 / 실행

- 패키징: `jpackage`로 Windows MSI, Linux deb/rpm
- Windows: Windows Service 등록, 시작 시 자동 실행
- Linux: systemd unit
- 첫 실행 시 `http://127.0.0.1:18080/setup`이 자동 열려 Pairing Wizard 시작
- Pairing Wizard 입력: `install_token`(사무소 admin이 발급), Cloud API URL, NAS root, agent_code

### 3.3 로컬 데이터 루트 (1차 결론 문서 §7.2 답습)

```text
%PROGRAMDATA%\ArchDox\agent\
  config/
    application.yml
    client.crt            # mTLS
    client.key
    truststore.jks
  state/
    agent.sqlite          # 로컬 인덱스 (job, photo meta cache)
  data/                   # <- NAS mount 또는 local
    offices/{officeCode}/
      reports/{reportId}/
        input/payload.json
        draft/             # LOCAL_ONLY draft mode의 step payload 정본
        photos/
          working/        # T1 (1920px)
          thumbnail/      # T2 (320px webp)
          original/       # T3, opt
        output/
          report.docx
          report.pdf
        logs/
      templates/
  logs/
    agent.log
    flower.log
```

### 3.4 NAS 연결

지원 프로토콜:
- Windows SMB (`\\nas\share`) — UNC path mount
- NFS (Linux)
- 로컬 디스크 (NAS 없는 사무소)

설정 항목:
```yaml
archdox:
  storage:
    kind: SMB       # SMB | NFS | LOCAL
    root: \\nas\archdox-data
    credentials:
      username: ${NAS_USER}
      password-ref: secret://nas-pass
    fallback-local: D:\ArchDoxFallback   # NAS 다운 시 임시 저장
```

NAS 다운 감지:
- 5초 timeout으로 path 검증 cron (1분 주기)
- 실패 시 `LocalNasUnreachableEvent` Bloom 발행 → fallback-local로 임시 저장 + Cloud에 status `DEGRADED` 보고
- 복구되면 fallback → NAS 자동 옮김 (Flower flow `nas-recovery-sync`)

### 3.5 업로드/드래프트 보안 경계

- Agent↔Cloud는 mTLS + service JWT로 인증한다.
- Client↔Agent 직접 사진 업로드는 Cloud가 발급한 signed upload token만 받는다. 브라우저/PWA에 client certificate를 설치하지 않는다.
- 기본 draft 정본은 Cloud 암호화 저장이지만, `LOCAL_ONLY` 사무소에서는 `data/offices/{code}/reports/{reportId}/draft/`가 상세 payload 정본이다.
- `LOCAL_ONLY` 모드에서도 Cloud에는 step별 저장 시각, revision, payload hash, local draft logical key를 보고한다.

### 3.6 문서 생성

`DocumentRenderFlow` (`03` 문서 §8.2):
- 템플릿 엔진: **docx4j** (Apache License 2.0 안전)
- PDF 변환: **LibreOffice headless** 우선, 실패 시 fallback to `docx4j` PDF export
- 이미지 삽입: 체크리스트 항목별 그룹 → 2x2 grid or 1행
- 폰트: 사무소 공통 한글 폰트(나눔/맑은고딕) 임베드 옵션

라이선스 회피:
- iText 5+, Aspose 등 상용 라이선스 제품은 기본 미사용
- 필요 시 사무소가 자체 라이선스 보유한 변환기를 plug-in (interface로 추상화)

## 4. 사내 관리 UI — 결정

### 4.1 비교

| 옵션 | 장점 | 단점 |
|---|---|---|
| **ArchDox Agent 내장 React 페이지 (localhost)** | 별도 배포·업데이트 채널 0, Cloud admin과 동일 컴포넌트 재사용, 원격 지원시 화면 공유 가능 | 사무소 직원이 브라우저 열어야 함 |
| C# WinForms (트레이 앱) | Windows 친화 UX, 트레이 알림 자연스러움 | C# 별도 코드베이스/배포, 업데이트 채널 필요, Linux 미지원 |
| C# WebForms | 1세대 기술, 신규 채택 부적합 | (논외) |
| Electron 별도 데스크탑 앱 | 트레이 + 자체 UI | 200MB 번들, 업데이트 채널 필요 |

### 4.2 결정: ArchDox Agent 내장 React 페이지 + (선택) 트레이 helper

- React UI는 `localhost:18080/admin`으로 접근. Agent와 같은 프로세스가 서빙.
- 트레이 아이콘이 필요하면 Java AWT `SystemTray` 사용 (가벼움) 또는 별도 50KB 짜리 Go/Rust helper.
- 사무소 IT 담당자는 브라우저 즐겨찾기로 충분. 일반 직원은 Cloud 모바일 앱만 사용.

### 4.3 관리 UI 기능

1. **상태 대시보드**: NAS 연결, Cloud 연결, 최근 heartbeat, 진행 중 job, 디스크 여유
2. **설정**: NAS 경로 / 자격증명, 프린터, AI endpoint, 자동 업데이트 on/off
3. **템플릿 관리**: Cloud에서 푸시된 템플릿 목록, 사무소 커스텀 템플릿 업로드
4. **작업 로그**: 최근 N건의 문서 생성 로그, 실패 사유, 재시도 버튼
5. **사진 캐시 관리**: T1/T2/T3 용량, 보존 정책, 수동 purge
6. **테스트 도구**: NAS write test, printer test, AI ping, Cloud ping
7. **페어링 / 마이그레이션**: re-pairing, 새 client cert 발급, 사무소 변경(드물게)
8. **백업**: agent.sqlite export, 설정 export

### 4.4 보안
- `localhost` binding 강제. `0.0.0.0` 변경 금지 (코드로 차단).
- 관리 UI 진입 시 사무소 관리자 password (Cloud의 OFFICE/ADMIN role 검증 또는 사전 설정 PIN)
- 모든 설정 변경은 Cloud `audit_logs`에도 비동기 동기화

## 5. NAS ↔ ArchDox Agent ↔ Cloud 데이터 흐름 요약

```text
NAS  <-- read/write --> ArchDox Agent  <-- mTLS REST --> Cloud API
                              |
                              v
                          Printer
                              |
                              v
                         AI Service
```

원본은 NAS에만, Cloud는 logical reference + thumbnail 만 (`04` 문서 §3).

## 6. 시간 동기화
- NTP 강제 (Windows: w32time, Linux: chrony)
- Cloud와의 시간 skew > 60s면 pairing/heartbeat 거부 + 관리 UI에 경고

## 7. 버전 호환
- Cloud API는 응답 헤더 `X-Agent-Min-Version` 강제
- Agent는 시작 시 Cloud `/agent/api/v1/handshake` 호출 → 자신이 미달이면 업데이트 안내 모드 (read-only)
- 자동 업데이트는 운영 phase 7 이후 (`08` 문서)

## 8. 백업 & 재해복구
- NAS 백업은 사무소 책임 (RAID + 외장 백업 권장). Agent 가이드 문서로 안내.
- `agent.sqlite`는 일 1회 `state/backup/agent-{yyyymmdd}.sqlite`로 export, 7일 retention
- 페어링 정보(client.key 등)는 사무소 관리자가 secure backup (Cloud는 fingerprint만 알 뿐 복구 불가)

## 9. 운영 모드
| 모드 | 동작 |
|---|---|
| `NORMAL` | 정상 |
| `READ_ONLY` | 신규 job 수신 차단, 진행 중인 것만 마무리. 라이선스 만료/버전 부적합 시 자동 진입 |
| `MAINTENANCE` | NAS 마이그레이션 등, 수동 |
| `OFFLINE` | Cloud 연결 끊김. 로컬 cache에 명령 큐잉, 복구 시 재동기화 |

## 10. 한 줄 요약
**메인 Agent = Java 21 + Spring Boot**, **관리 UI = localhost 내장 React 페이지**. C#은 도입하지 않는다. 도입한다면 Windows-only 보조 helper로 한정한다.
